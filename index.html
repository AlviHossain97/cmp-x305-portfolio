    02 / Week 5

    Web Application Security Lab (WPScan)

    Application-layer reconnaissance against a WordPress instance to map the full attack surface: themes, plugins, resources, and credential exposure.

    🎯
    Key Takeaway
    Main analytical point
    WPScan works at the application layer, not just the network layer. It flagged the active theme version, an Akismet advisory requiring version validation, exposed resources including robots.txt and XML-RPCXML Remote Procedure Call — a WordPress API endpoint that can be abused for brute-force and DDoS attacks, and demonstrated how quickly reconnaissance can progress to a brute-force login attempt. Each finding needed manual version confirmation before being treated as a confirmed vulnerability — that distinction between flagged and confirmed is the core analytical lesson of this lab.
    AimMethodologyEvidenceFindingsMitigationsReflection

    Aim

    The aim of this lab was to use WPScan from Kali Linux in order to assess a WordPress instance at http://192.168.123.30 and identify any exposed themes, plugins and resources, then interpret the security risks from the results [2], [3]. This lab addressed LO1Learning Outcome 1 — Identify vulnerabilities and recommend hardening actions because I was able to translate the scan output into concrete hardening actions, and LO3Learning Outcome 3 — Assess privacy and data-protection implications because it showed how publicly exposed WordPress components can reduce user privacy.

    Methodology

    The target WordPress server was hosted at 192.168.123.30 and WPScan ran from a Kali Linux VM. I followed the structure of the lab sheet rather than making up my own order because the sequence builds on itself:

    1. Update the WPScan database before scanning;
    2. run a fuller WordPress scan with API-backed enrichment and aggressive plugin checks;
    3. capture theme and plugin enumeration output;
    4. execute the lab brute-force task with the discovered username and a supplied password list; and
    5. review exposed resources and vulnerability output for practical attack paths.

    That sequence mattered because the later steps depended on the earlier ones. Enumeration revealed what WordPress components were present, the API-backed scan added vulnerability intelligence on top of that, and the brute-force exercise showed how quickly reconnaissance can move toward credential abuse if login protections are weak.

    Findings and Analysis

    For me the most useful result was not one single finding but the way WPScan assembled the application's attack surface. Figure W5-3 identified the active theme and its version, and then Figure W5-7 revealed robots.txt, XML-RPC, the readme and discoverable uploads. Together these gave me a checklist of entry points to document and address.

    Plugin enumeration added more context to the scan. Figure W5-6 showed akismet and ninja-forms, and WPScan attached an advisory for an unauthenticated stored XSS in akismet. That advisory only came through because I ran the scan with the API token. Without the token the scanner would only detect which plugins are installed; with the token it cross-references versions against WPVulnDB and returns matched advisories. The advisory is a lead — it indicates a potentially vulnerable component that still requires version confirmation before it can be treated as a confirmed finding.

    The lab brute-force task was also important to include because it showed the progression from discovery to attempted account abuse. Figure W5-4 shows WPScan performing a dictionary attack against the login endpoint. If a WordPress site has no lockout policy then automated password guessing has unlimited attempts, and the enumeration work done in the earlier steps feeds directly into which username to target.

    Figure W5-7 was the strongest single reconnaissance screenshot. robots.txt exposed administrative paths, XML-RPC was enabled, readme.html was public and several upload directories were browsable. These reconnaissance findings lower the effort required for every subsequent attack stage.

    This lab helped cover the CIA model. Exposed files and paths affect confidentiality by disclosing information that makes later attacks easier. The stored XSS advisory, if confirmed exploitable, could affect integrity by modifying page content. And sustained brute-force via XML-RPC could affect availability. This section addresses LO1 directly — the findings translate into concrete hardening actions — and LO3 because WordPress component disclosure reduces user privacy.

    Recommended Mitigations

    • Patch WordPress core, themes, and plugins regularly; remove components that are not needed
    • Disable directory listing on the web server (e.g. Options -Indexes on Apache)
    • Remove or restrict access to readme.html and similar deployment artefacts
    • Disable xmlrpc.php unless there is a specific business need for it
    • Apply login rate limiting, strong password policy, and MFA to administrator accounts
    • Review the uploads area and remove unnecessary public file exposure
    • Verify flagged plugin advisories against exact installed versions before prioritising patching

    Reflection

    WPScan was useful because it worked at the application layer rather than just the network layer. A general port scan might tell me that 80/tcp is open but WPScan told me which WordPress theme and plugins were installed, whether they had known advisories and what resources were discoverable from the outside. That level of detail is what made the lab findings actionable rather than just theoretical.

    The main limitation was confidence. WPScan flagged a potentially risky plugin but I still had to check whether the installed version fell within the affected range. That distinction between a flagged advisory and a confirmed vulnerability is something I would handle more carefully next time by recording the exact installed version for every flagged component at the point of scanning rather than relying on the advisory reference alone.

    03 / Week 7

    Juice Shop Lab

    Hands-on exploitation of intentionally vulnerable web application behaviour across ten challenges, with per-challenge broken-control analysis.

    ⚡
    Key Takeaway
    Main analytical point
    Juice Shop was the most analytically useful lab in the portfolio because it forced me to think like an attacker. The ten challenges documented here are not random — they cluster around information disclosure, client-side injection, broken access control, and weak authentication, which maps directly onto the OWASP Top Ten. No single challenge was dramatic on its own, but the chain from hidden routes to directory exposure to admin login to privileged data manipulation shows how ordinary oversights compound into serious risk. This lab provides the strongest evidence for LO1, LO2, and LO3 in the portfolio.
    AimMethodologyCoverageEvidenceFindingsMitigationsReflection

    Aim

    The aim of this lab was to explore intentionally vulnerable web application behaviour in Juice Shop, complete challenges with screenshot evidence and explain the security failure behind each one [4], [5]. This lab addressed LO1, LO2Learning Outcome 2 — Apply cryptographic and authentication principles and LO3 because it covered injection, authentication bypass, session handling, client-side trust and directory exposure.

    Methodology

    I started Juice Shop locally on the lab VM on port 3000. After confirming the application loaded successfully I worked through the challenges one at a time and took a screenshot whenever a challenge flag appeared.

    I approached the lab by asking myself what security control had failed in each case. I felt this produced a better analysis than simply collecting green completion banners because it forced me to explain not just what I did but why it worked.

    Challenge Coverage

    Findings and Analysis

    The first theme I noticed was hidden functionality. The Score Board challenge in Figure W7-3 showed that the application trusted obscurity too much. If a route exists in the client-side JavaScript and the only protection is that it is not linked in the navigation, an attacker can still reach it just by reading the page source. The same idea applies to the /ftp/ directory in Figure W7-6 — the files were not linked anywhere but were still served without access control.

    The DOM XSS and Bonus Payload challenges in Figures W7-4 and W7-5 showed how dangerous client-side trust can be. The browser executed attacker-controlled HTML because the application inserted user input into the DOM without encoding or sanitisation. In terms of LO1 the fix is output encoding and Content Security Policy. In terms of LO3, persistent XSS could steal session tokens or personal data — both a privacy risk and an integrity risk.

    The strongest admin-related chain was Login Admin, Admin Section and Five-Star Feedback in Figures W7-9 to W7-11. Once privileged access is established the attacker gains visibility into user data and the ability to modify it. Deletion of user-generated feedback demonstrates the integrity impact, and the same access level would expose email addresses or order details, extending the privacy consequence.

    The Password Strength challenge in Figure W7-12 is where LO2 needed more explicit treatment. Weak passwords are a problem even when a system uses one-way hashing because common passwords appear in pre-computed lists. Hashing protects stored passwords only if the original password has enough entropy to resist offline cracking.

    Login Admin also points toward session security. After authentication succeeds the platform still needs cryptographically unpredictable session and password-reset tokens, or even a patched login flow can be undermined by predictable session handling.

    Availability was easier to miss but it is present here too. The Error Handling challenge in Figure W7-8 shows unstable application behaviour. If deliberate bad input can provoke a crash or disclose a stack trace then more persistent fault injection could degrade the service entirely.

    This section addresses all three learning outcomes. It addresses LO1 by showing what controls failed and what needs to be fixed; LO2 by connecting hashing and session tokens to the Password Strength and Login Admin challenges; and LO3 by showing how directory exposure, admin access and XSS all create privacy consequences. The cumulative effect across all ten challenges makes this lab the strongest demonstration that LO3 is not a single-issue concern.

    Recommended Mitigations

    • Remove or properly gate hidden routes so administrative or diagnostic pages are not reachable without server-side authorisation
    • Apply output encoding, DOM sanitisation, and a restrictive Content Security Policy to reduce XSS risk
    • Disable public directory listings and move sensitive files outside web-accessible paths
    • Restrict monitoring and metrics endpoints to authenticated or internal users only
    • Enforce strong password policy, MFA, rate limiting, and lockout controls on privileged accounts
    • Ensure session and password-reset tokens are cryptographically unpredictable and rotated appropriately
    • Log privileged actions such as feedback deletion so integrity-impacting changes can be traced and reviewed

    Reflection

    This was the most useful lab in the portfolio for me because it forced me to think like an attacker rather than like a tool operator. WPScan and Nessus are good at producing lists of issues but Juice Shop made me exploit those issues step by step and understand why each control failed. That process gave me a much better understanding of how vulnerabilities connect to each other.

    What I learnt most clearly was that insecure web applications rarely fail in one dramatic way. They usually fail through a series of ordinary mistakes, none of which look critical on their own, but which chain together into something serious. Juice Shop changed how I think about chained risk because no single challenge was dramatic on its own but the combination of disclosure, injection and broken access control created a much larger attack surface than any one weakness suggested.

    04 / Week 8

    Vulnerability Assessment Lab (Nmap and SSH)

    Network-level host and service discovery with Nmap, followed by secure administration using SSH key-based authentication, tunnelling, and multiplexing.

    🔐
    Key Takeaway
    Main analytical point
    This lab worked because the two halves supported each other. Nmap showed what an attacker could see from the outside; SSH showed the right way to administer the same exposed systems. Without the discovery phase, SSH hardening would have been done in isolation without understanding the threat model. Without the SSH exercises, the Nmap findings would have stopped at a list of ports. This lab changed how I think about the relationship between discovery and administration — knowing what is exposed is only useful if you also know how to lock it down properly.
    AimMethodologyNmap EvidenceNmap FindingsSSH EvidenceSSH FindingsMitigationsReflection

    Aim

    The aim of this lab was to assess a small lab network using Nmap and then administer the discovered systems securely using SSH [6]–[9]. It addressed LO1 because I identified exposed services and translated the scan output into risk-based recommendations, and LO2 because SSH was a practical demonstration of confidentiality, integrity and authentication through cryptographic primitives.

    Methodology

    I treated the lab as two connected tasks rather than two separate tool demonstrations.

    For the Nmap portion I carried out host discovery, focused TCP scans, UDP scanning, service and OS detection, NSE vulnerability checks, traceroute, IPv6 scanning and several advanced options including decoys, fragmentation and ACK scans.

    For the SSH portion I connected to the gateway, created a 4096-bit RSA key pair, deployed the public key, tested SCP, used both local and remote port forwarding, configured ControlMaster multiplexing and confirmed passwordless key-based login.

    SSH Findings and Analysis

    🔐SSH Security Properties3 properties
    Security Property What SSH Provided in This Lab Residual Risk if Mishandled
    Confidentiality Encrypted administration sessions, SCP transfers, and forwarded traffic Weakens quickly if keys are stolen or tunnels are exposed carelessly
    Integrity Host verification and authenticated sessions reduce risk of tampering in transit Trust depends on key protection and correct host validation
    Availability Secure remote administration without exposing every internal service directly Overuse of tunnelling or broken key management can create operational fragility

    This part of the lab made LO2 much more concrete than a theoretical discussion of cryptography would have. SSH protects confidentiality by encrypting administrative traffic, integrity through MACs that detect tampering, and authentication through key-based identity. Using these properties in practice rather than just describing them made the learning outcome much clearer.

    Generating a 4096-bit RSA key pair and deploying it with ssh-copy-id showed why the full chain matters — a strong key delivered to the correct authorized_keys file, verified by a subsequent passwordless login. Key size alone is not enough if the deployment step is careless.

    SCP and the tunnelling exercises in Figures W8-13 to W8-15 showed that SSH can carry file transfers, forward local ports, or expose a local service to the remote side. All three are useful for legitimate administration but all three could also be abused to bypass network controls if SSH access is not properly governed. This section addresses LO1 by identifying which services should be reduced or segmented, and LO2 by showing how SSH's cryptographic primitives protect every operation in the evidence.

    Recommended Mitigations

    • Restrict SSH, NRPE, MySQL, and unnecessary UDP services to only the hosts or interfaces that genuinely need them
    • Apply the same service-hardening decisions to IPv6 exposure as to IPv4 exposure
    • Treat NSE findings as leads for validation rather than automatic proof of compromise
    • Prefer key-based SSH authentication over reusable passwords and protect private keys with strong local controls
    • Monitor and govern SSH tunnelling so legitimate administrative use does not silently bypass network boundaries
    • Document which advanced scan techniques produced reliable evidence and which did not, so future assessments start from the strongest methods

    Reflection

    My main takeaway was that secure administration is not just about having SSH available. It is about using it deliberately: keys instead of reusable passwords, tunnelling scoped to what is needed rather than left open by default, and multiplexing for operational efficiency. What I would do differently next time is document which scan techniques produced reliable evidence and which did not, so that future assessments start from the strongest methods instead of repeating everything.

    Nmap and SSH changed how I think about the relationship between discovery and administration because knowing what is exposed is only useful if you also know how to lock it down properly.

    Reflection

    Nessus was the clearest tool in the portfolio for turning technical exposure into prioritised management output. Compared with Nmap which is good at discovery but leaves prioritisation to the analyst, Nessus produces structured severity ratings, remediation guidance and evidence that is already formatted for reporting.

    Its limitation is also what makes it useful: it abstracts aggressively. That helps with prioritisation but it can tempt an analyst into treating every medium finding as identical when in reality the operational context matters more than the severity label. The IP forwarding finding showed that clearly because a medium rating on a host whose job is to route traffic means something different from the same rating on a host that should not be forwarding at all. That is the kind of judgement that Nessus cannot make for you, and learning to apply it is what I will take forward from this lab.

    06 / Overall

    Overall Reflection

    Cross-lab learning, tool effectiveness, and what I would do differently next time.

    💡
    Overarching Lesson
    Main analytical point
    Each tool in this portfolio does a different job — and understanding where each one stops is as important as understanding what it can do. WPScan finds leads that need validation. Juice Shop shows how controls fail in combination. Nmap maps the surface. SSH secures the access to what Nmap found. And Nessus prioritises — but only if you apply the context that the scanner cannot supply itself.

    Each lab taught me a different part of the same lesson.

    WPScan showed me how quickly application-specific reconnaissance can expose useful information. Juice Shop showed me what those kinds of weaknesses look like from the inside. Nmap and SSH linked network-level discovery with secure management. And Nessus turned all of that into structured severity-based output that can be used for prioritisation and reporting.

    What I would do differently next time is capture even more comparative evidence while I am in the lab instead of relying on cleanup later. The extra Juice Shop analysis chart and the Nessus severity comparison were the most valuable additions to the write-up, and I could have produced even more of those if I had planned the visuals from the start.

    WPScan changed how I think about version validation because it showed me that a plugin advisory is only a lead until you confirm the installed version matches the affected range. Juice Shop changed how I think about chained risk because no single challenge was dramatic on its own but the combination of disclosure, injection and broken access control created a much larger attack surface than any one weakness suggested. Nmap and SSH changed how I think about the relationship between discovery and administration because knowing what is exposed is only useful if you also know how to lock it down properly. And Nessus changed how I think about prioritisation because the severity label alone did not tell me which finding mattered most without understanding the host role and network context.

    The biggest overall learning point was that security tools do different jobs. WPScan and Nmap are strong at discovery, SSH is about secure operation and Nessus is about triage and prioritisation. Knowing what each tool is for and where its limits are is what turns technical output into a useful security assessment.
