In this portfolio I have documented four practical cyber security labs that I completed for CMP-X305. These were a WordPress assessment using WPScan, an OWASP Juice Shop challenge series, a combined Nmap and SSH exercise, and a Nessus vulnerability scan. Together these labs cover discovery, exploitation, secure administration and automated assessment.

All of the activity took place inside an isolated virtual lab that used Kali Linux and intentionally vulnerable systems, which is important because it means nothing was done against real-world targets.

Something I noticed across all four labs was that small weaknesses become much more serious when you combine them. Public files help with reconnaissance, reconnaissance helps with exploitation, and the exploitation itself often just relies on ordinary oversights rather than anything exotic.

02 / Week 5

Web Application Security Lab (WPScan)

Application-specific reconnaissance against a WordPress instance to identify exposed themes, plugins and resources.

Aim

The aim of this lab was to use WPScan from Kali Linux in order to assess a WordPress instance at http://192.168.123.30 and identify any exposed themes, plugins and resources, then interpret the security risks from the results [2], [3]. This lab addressed LO1 because I was able to translate the scan output into concrete hardening actions, and LO3 because it showed how publicly exposed WordPress components can reduce user privacy.

Environment and Methodology

The target WordPress server was hosted at 192.168.123.30 and WPScan ran from a Kali Linux VM. I followed the structure of the lab sheet rather than making up my own order because the sequence builds on itself:

  1. update the WPScan database before scanning;
  2. run a fuller WordPress scan with API-backed enrichment and aggressive plugin checks;
  3. capture theme and plugin enumeration output;
  4. execute the lab brute-force task with the discovered username and a supplied password list; and
  5. review exposed resources and vulnerability output for practical attack paths.

That sequence mattered because the later steps depended on the earlier ones. Enumeration revealed what WordPress components were present, the API-backed scan added vulnerability intelligence on top of that, and the brute-force exercise showed how quickly reconnaissance can move toward credential abuse if login protections are weak.

Findings and Analysis

For me the most useful result was not one single vulnerability but the way WPScan put together the application's attack surface. Figure W5-3 identified the active theme and its version, and then Figure W5-7 revealed robots.txt, XML-RPC, the WordPress readme and discoverable uploads. When I combined all of these they gave me a checklist of entry points that I could document and address.

Plugin enumeration added more context to the scan. Figure W5-6 showed akismet and ninja-forms, and WPScan attached an advisory for an unauthenticated stored XSS in akismet. That advisory only came through because I ran the scan with the API token. Without the token the scanner would only detect which plugins are installed, but with the token it cross-references versions against WPVulnDB and returns matched CVEs.

The lab brute-force task was also important to include because it showed the progression from discovery to attempted account abuse. Figure W5-4 shows WPScan performing a dictionary attack against the login endpoint. Even though I was using a lab wordlist for this, the underlying problem is the same regardless because if a WordPress site has no lockout policy then automated password guessing has unlimited attempts.

Figure W5-7 was the strongest single reconnaissance screenshot. robots.txt exposed administrative paths, XML-RPC was enabled, readme.html was public and several upload directories were browsable. That kind of information would help an attacker plan further steps without needing to run extra scans.

From a defensive perspective what I found more interesting was the recon-to-risk chain. Theme and plugin disclosure give an attacker specific components to check against the NVD. Exposed paths reduce the effort needed to find administrative or upload interfaces. Together all of these reconnaissance findings lower the effort required for the next stage of exploitation.

This lab also helped me cover the CIA model. The exposed files and paths affect confidentiality because they disclose information that could make later attacks easier. The stored XSS advisory, if exploitable, could affect integrity by modifying page content in the browser. And the potential for sustained brute-force attempts via XML-RPC affects availability because enough requests could degrade or deny the service entirely.

This section also addresses LO1 because the findings translate directly into measures that can be applied on the host. In practical terms the first response to this evidence would be to restrict XML-RPC, remove the readme, review plugin update status, enforce login protection and block directory listing.

Based on the evidence I collected, the three most actionable issues are:

  1. restrict XML-RPC and exposed resource disclosure because they widen both reconnaissance and availability risk from a single configuration state;
  2. verify and patch plugin exposure, especially the Akismet advisory, because application-layer vulnerabilities can turn disclosure into script execution or content manipulation; and
  3. harden the login path with rate limiting, MFA, and stronger passwords because the brute-force task shows how quickly the assessment can move from discovery into account attack activity.

Recommended Mitigations

  • Patch WordPress core, themes, and plugins regularly, and remove components that are not needed
  • Disable directory listing on the web server, for example with Options -Indexes on Apache
  • Remove or block access to readme.html and similar deployment artefacts
  • Disable xmlrpc.php if there is no business need for it, or restrict it to trusted clients only
  • Apply login rate limiting, strong password policy, and multi-factor authentication to administrator accounts
  • Review the uploads area for sensitive files and remove unnecessary public exposure
  • Verify flagged plugin advisories manually before prioritising patching work

Reflection

WPScan was effective because it worked at the application layer. A general network scan might tell me that 80/tcp is open but WPScan told me which WordPress theme and plugins were installed, whether they had known advisories and what resources were discoverable from the outside. That level of detail is what made the lab findings practical rather than just theoretical.

The main limitation was confidence though. WPScan can point me toward a risky plugin or configuration but I still have to check whether the installed version is actually vulnerable or not. Something I would do differently next time would be to record the installed version for every flagged component at the time of scanning rather than relying on the advisory reference alone.

03 / Week 7

Juice Shop Lab

Hands-on exploitation of intentionally vulnerable web application behaviour with challenge-by-challenge analysis.

Aim

The aim of this lab was to explore intentionally vulnerable web application behaviour in Juice Shop, complete challenges with screenshot evidence and explain the security failure behind each one [4], [5]. This lab addressed LO1, LO2 and LO3 because it covered injection, authentication bypass, session handling, client-side trust and directory exposure.

Environment and Methodology

I started Juice Shop locally on the lab VM and it was made available in the browser on port 3000. After I confirmed the application loaded successfully I worked through the challenges one at a time and took a screenshot whenever a challenge flag appeared.

I approached the lab by asking myself what security control had failed in each case. I felt this produced a better analysis than simply collecting green completion banners because it forced me to explain not just what I did but why it worked.

Findings and Analysis

The first theme I noticed was hidden functionality. The Score Board challenge in Figure W7-3 showed that the application trusted obscurity too much. If a route exists in the client-side JavaScript and the only protection is that it is not linked in the navigation, an attacker can still reach it just by reading the page source. The same idea applies to the /ftp/ directory in Figure W7-6 because the files were not linked anywhere on the site but they were still served by the web server since there was no access control in place.

The DOM XSS and Bonus Payload challenges in Figures W7-4 and W7-5 showed me how dangerous client-side trust can be. The browser executed attacker-controlled HTML because the application inserted user input into the DOM without any encoding or sanitisation. In terms of LO1 the fix is output encoding and Content Security Policy. In terms of LO3, persistent XSS could steal session tokens or personal data so it is both a privacy risk and an integrity risk.

The strongest admin-related chain was Login Admin, Admin Section and Five-Star Feedback in Figures W7-9 to W7-11. Once privileged access exists the attacker gains both visibility into user data and the ability to modify it. Deletion of user-generated feedback proves the integrity impact, and if the same access lets the attacker view email addresses or order details then the privacy impact follows as well.

The Password Strength challenge in Figure W7-12 is where I felt LO2 needed more explicit treatment. Weak passwords are a problem even when a system uses one-way hashing because common passwords appear in pre-computed rainbow tables or dictionary lists. Hashing protects stored passwords only if the original password has enough entropy to resist offline cracking.

Login Admin also points toward session security. After authentication succeeds the platform still needs cryptographically unpredictable session tokens and password-reset tokens. Otherwise, even a patched login flow can be undermined by predictable session handling.

Availability was easier to miss at first but it is present here as well. The Error Handling challenge in Figure W7-8 shows unstable application behaviour. If deliberate bad input can provoke a crash or disclose a stack trace then more persistent fault injection could degrade the service entirely.

This section addresses all three learning outcomes directly. It addresses LO1 by showing what controls failed and what needs to be fixed, LO2 by connecting hashing and session tokens to the Password Strength and Login Admin challenges, and LO3 by showing how directory exposure, admin access and XSS all create privacy consequences.

Figure W7-14 makes the overall pattern easier to see because the documented challenges were not random. They clustered around information disclosure, client-side injection, broken access control and weak authentication. That clustering is consistent with the OWASP Top Ten which is why this lab was valuable as a practical illustration of real-world vulnerability categories.

This week is also the clearest evidence for LO3 in the whole portfolio because it shows privacy loss from several angles at once. Hidden route discovery exposes the entire challenge list which is information disclosure, /ftp/ exposes confidential documents which is a file-level data leak, SQL injection grants access to user records which is authentication bypass leading to data exposure, and admin access allows data deletion which is an integrity violation. The cumulative effect makes this lab the strongest demonstration that LO3 is not a single-issue concern.

Recommended Mitigations

  • Remove or properly gate hidden routes so administrative or diagnostic pages are not reachable without server-side authorisation
  • Apply output encoding, DOM sanitisation, and a restrictive Content Security Policy to reduce XSS risk
  • Disable public directory listings and move sensitive files outside web-accessible paths
  • Restrict monitoring and metrics endpoints to authenticated or internal users only
  • Enforce strong password policy, MFA, rate limiting, and lockout controls on privileged accounts
  • Ensure session and password-reset tokens are cryptographically unpredictable and rotated appropriately
  • Log privileged actions such as feedback deletion so integrity-impacting changes can be traced and reviewed

Reflection

This was the most useful lab in the portfolio for me because it forced me to think like an attacker rather than like a tool operator. WPScan and Nessus are good at producing lists of issues but Juice Shop made me exploit those issues step by step and understand why each control failed. That process gave me a much better understanding of how the vulnerabilities connect to each other.

What I learnt most clearly was that insecure web applications rarely fail in one dramatic way. They usually fail through a series of ordinary mistakes and none of which look critical on their own but they chain together into something serious.

04 / Week 8

Vulnerability Assessment Lab (Nmap and SSH)

Network-level discovery with Nmap followed by secure administration using SSH key-based authentication and tunnelling.

Aim

The aim of this lab was to assess a small lab network using Nmap and then administer the discovered systems securely using SSH [6]-[9]. It addressed LO1 because I identified exposed services and translated the scan output into risk-based recommendations, and LO2 because SSH was a practical demonstration of confidentiality, integrity and authentication through cryptographic primitives.

Methodology

I treated the lab as two connected tasks rather than two separate tool demonstrations.

For the Nmap portion I carried out host discovery, focused TCP scans, UDP scanning, service and OS detection, NSE vulnerability checks, traceroute, IPv6 scanning and several advanced options including decoys, fragmentation and ACK scans.

For the SSH portion I connected to the gateway, created a 4096-bit RSA key pair, deployed the public key, tested SCP, used both local and remote port forwarding, configured ControlMaster multiplexing and confirmed passwordless key-based login.

Host discovery identified three live systems on the 192.168.123.0/24 range which were 192.168.123.1, 192.168.123.10 and 192.168.123.30 (Figure W8-1). The directed TCP scans in Figure W8-2 separated those hosts by their open port profile. Together these first two steps established the target list and gave structure to everything that followed.

The gateway at 192.168.123.1 exposed 22/tcp and 5666/tcp while the WordPress server at 192.168.123.30 exposed 22/tcp, 80/tcp and 3306/tcp (Figure W8-2). Aggressive scanning in Figure W8-4 added version information, OS guesses and default-script output for the WordPress host. These are the kind of details that an attacker would collect in a real engagement before trying to exploit anything.

UDP results in Figure W8-3 widened that picture by showing several open or filtered services on the gateway including DNS, DHCP, NTP and SNMP-related ports. Those are often ignored in TCP-only assessments but they can carry their own risks, for example SNMP with a default community string.

The NSE results in Figure W8-5 were useful but needed caution. Nmap reported possible SQL injection and CSRF issues against the WordPress service but these are scripted checks and not confirmed exploits. They still helped because they showed me where to investigate further.

The advanced options in Figures W8-6 and W8-7 were valuable because they showed me that scan behaviour can be adapted. Decoys and fragmentation are designed to complicate IDS detection and ACK scanning maps filtering rules instead of open ports. These techniques are not always needed but knowing they exist means I can adjust my approach if a target environment is actively monitored.

IPv6 scanning in Figure W8-9 was also a good reminder that the attack surface may exist on more than one protocol stack. If defenders only harden IPv4 and leave IPv6 controls open then the dual-stack host becomes a bypass.

This part of the lab made LO2 much more concrete for me than a theoretical discussion of cryptography would have. SSH protects confidentiality by encrypting administrative traffic, integrity through MACs that detect tampering, and authentication through key-based identity. Using these properties in practice rather than just describing them in theory made the learning outcome much clearer.

Generating a 4096-bit RSA key pair and deploying it with ssh-copy-id also showed me why cryptographic primitives matter in practice. The benefit does not come from the key size alone but from the whole chain, which is a strong key delivered to the right authorized_keys file, verified by a subsequent passwordless login and tested under different conditions.

SCP and the tunnelling exercises in Figures W8-13 to W8-15 were also useful from a defensive perspective. They proved that SSH can carry file transfers, forward local ports through a remote host or expose a local service to the remote side. All three are useful for legitimate administration but all three could also be used to bypass network controls if SSH access is not properly governed.

This section addresses LO1 by identifying which services are exposed and should be reduced or segmented, and LO2 by demonstrating how SSH uses cryptographic primitives to protect confidentiality, integrity and authentication across every operation shown in the evidence.

Recommended Mitigations

  • Restrict SSH, NRPE, MySQL, and any unnecessary UDP services to only the hosts or interfaces that genuinely need them
  • Apply the same service-hardening decisions to IPv6 exposure as to IPv4 exposure
  • Treat NSE findings as follow-up tasks for validation rather than automatic proof of compromise
  • Prefer key-based SSH authentication over reusable passwords and protect private keys with strong local controls
  • Monitor and govern SSH tunnelling so legitimate administrative use does not silently bypass network boundaries
  • Document which advanced scan techniques produced reliable evidence and which did not

Reflection

This lab worked well because the two halves supported each other. Nmap showed me what an attacker could see and SSH showed me the right way to manage those exposed systems. Without the discovery phase I would have been configuring SSH in isolation and without the SSH exercises the Nmap findings would have stopped at a list of ports.

My main takeaway was that secure administration is not just about having SSH enabled. It is about using it properly, so keys instead of reusable passwords, tunnelling done deliberately rather than accidentally, and multiplexing for operational efficiency.

Vulnerability Scanning and Mitigation Lab (Nessus)

Automated vulnerability scanning with structured severity-based output for prioritisation and remediation planning.

Aim

The aim of this lab was to install, configure and use Nessus in order to scan at least two virtual machines and interpret the resulting findings [10], [11]. This lab addressed LO1 because I identified host-level vulnerabilities and translated them into prioritised remediation steps, and LO2 because findings such as SSH weaknesses and IP forwarding relate to confidentiality, integrity and availability.

Methodology

I started the Nessus service on Kali Linux and connected to the interface at https://localhost:8834 where I completed the initial setup, selected the Basic Network Scan template and ran scans against two targets:

  • 192.168.123.1 — Master Gateway
  • 192.168.123.30 — Ubuntu WordPress

After launching both scans I reviewed the internal Nessus history screens, compared the severity counts and exported a report for each host. To add more depth I also created a side-by-side chart from the exported data.

Findings and Analysis

The internal Nessus screens in Figures W9-6 and W9-7 matter because they show that I directly used the scanner rather than just relying on exported PDFs. They demonstrate that I actually navigated the Nessus interface, monitored scan progress and reviewed history.

The Master Gateway report showed 0 critical, 0 high, 1 medium, 1 low and 24 informational findings. The most notable item was IP Forwarding Enabled. In a production network unnecessary packet forwarding can be misused for man-in-the-middle attacks or lateral movement, but in this lab context the gateway is expected to route traffic so the finding is more of a design decision rather than a misconfiguration.

The Ubuntu WordPress report showed 0 critical, 0 high, 1 medium, 1 low and 35 informational findings. The most important item was the Terrapin SSH vulnerability (CVE-2023-48795). This advisory affects SSH protocol negotiation which makes it directly relevant to LO2 and the cryptographic integrity of remote administration.

The comparison table and chart make one additional point clear which is that raw counts are not enough by themselves. The WordPress host had more informational findings but the Master Gateway finding about IP forwarding had more potential operational impact. Severity ratings need to be read alongside the role and exposure of the host.

This matters because severity totals are only a starting point. Nessus is useful for triage but prioritisation still depends on host role, exposure and the network context. That is why I included the comparison chart because it gives a visual baseline for risk prioritisation rather than relying on numbers alone.

Reflection

Nessus was the clearest tool in the portfolio for turning technical exposure into prioritised management output. Compared with Nmap which is excellent at discovery but leaves prioritisation to the analyst, Nessus produces structured severity ratings, remediation guidance and evidence that is already formatted for reporting.

Its weakness is also what makes it useful though because it abstracts aggressively. That is great for prioritisation but it can tempt an analyst into treating every medium finding as identical when in reality the operational context matters more than the colour code. The IP forwarding finding demonstrated that clearly.

06 / Overall

Overall Reflection

Cross-lab learning, tool effectiveness, and what I would do differently next time.

Each lab taught me a different part of the same lesson.

WPScan showed me how quickly application-specific reconnaissance can expose useful information. Juice Shop showed me what those kinds of weaknesses look like from the inside. Nmap and SSH linked network-level discovery with secure management. And Nessus turned all of that into a structured severity-based output that can be used for prioritisation and reporting.

What I would do differently next time is capture even more comparative evidence while I am in the lab instead of relying on cleanup later. The extra Juice Shop analysis chart and the Nessus severity comparison were the most valuable additions to the write-up, and I could have produced even more of those if I had planned the visuals from the start. I feel WPScan is really effective in telling us whether WordPress themes and plugins are vulnerable as it gives us the necessary information about the plugins and themes, so we know why they are vulnerable. As for Juice Shop, this was really effective in visualising the vulnerabilities themselves within a website and allowing us to actually exploit those vulnerabilities. Nmap and SSH showed us how ports can be scanned and so we can see what is actually exposed as well as how systems can be accessed via the hostname, IP address and credentials of a system. Nessus was effective at breaking down the vulnerabilities within a system in a structured way.

The biggest overall learning point was that security tools do different jobs. WPScan and Nmap are strong at discovery, SSH is about secure operation and Nessus is about triage and prioritisation. Knowing what each tool is for and where its limits are is what turns technical output into a useful security assessment.
